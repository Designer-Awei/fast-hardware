# Fast Hardware - 账号管理 PRD（8-account）

## 文档定位

- 本文件定义桌面端账号体系、登录入口、会话管理与个人中心界面规划。
- 本文优先服务当前开发阶段：`邮箱密码注册 / 登录 -> 个人中心 UI -> Supabase 接入层 -> 后续 OAuth 扩展`。
- 社区内容、审核、OSS 路径与创客集市发布流见 `feature-prd/9-maker-marketplace_prd.md`。
- 画布分享预览与导出见 `feature-prd/7-project-share_prd.md`。

---

## 1. 目标

1. 在桌面端提供立即可用的基础账号入口，先支持 **邮箱密码注册 / 登录**。
2. 在主导航中增加稳定、低打扰的登录入口，让用户随时能查看登录态与进入个人中心。
3. 通过 `Supabase Auth + profiles` 建立统一身份模型，为后续点赞、收藏、上传、审核打基础。
4. 让个人中心同时承载三类内容：账号设置、我的本地方案、创作中心数据入口。
5. 为后续 OAuth（GitHub / Gitee 等）预留扩展位，但不阻塞首版上线。

---

## 2. 首版范围

### In Scope

- 顶部导航新增圆形用户头像，显示当前登录态
- 未登录点击头像可打开登录 / 注册窗口
- 新增「个人中心」一级标签页
- 个人中心二级标签页：
  - `账号设置`
  - `我的项目`（原「我的方案」命名；本地项目卡片 + 云端备份能力）
  - `创作中心`
- `账号设置` 内支持：
  - 未登录时显示一体化登录页
  - 已登录时显示当前账号状态与设置
  - 邮箱密码注册
  - 邮箱密码登录
  - 退出登录
- `我的项目` 读取系统设置中的项目文件夹路径，将本地项目以卡片形式展示；已登录时同步云端备份状态并支持上传/更新、撤销、下载恢复（见第 4.3、6.4 节）
- `创作中心` 先搭界面骨架，预留上传统计、点赞、收藏、复刻指标
- 管理员登录后，在个人中心下额外显示 `社区管理` 二级标签页
- 主进程维护 Supabase 会话，支持应用重启后恢复登录态

### Out of Scope（本轮不做）

- GitHub / Gitee / 飞书 / 微信 OAuth 真正接入
- 手机号短信登录
- 多因素认证（MFA）
- **全自动**跨设备项目同步、冲突合并与后台增量同步（个人中心已提供 **有限容量** 的「整包上传 / 撤销 / 下载恢复」，不等同于完整云同步产品）
- 管理员审核后台完整实现
- 上传 OSS 与内容审核实际链路

## 3. 用户故事

1. 作为未登录用户，我可以在右上角看到头像入口，点击后用邮箱和密码注册 / 登录。
2. 作为已登录用户，我希望头像能直观看到当前登录状态，并能一键进入个人中心。
3. 作为用户，我希望在个人中心查看我的本地项目卡片，而不是重新去文件夹里翻找。
4. 作为创作者，我希望提前看到创作中心的数据结构与入口，后续方便接入创客集市。
5. 作为普通用户，我希望应用重启后仍保留登录态，不必重复登录。
6. 作为已登录用户，我希望对指定本地项目执行云端备份的上传/更新、撤销与下载恢复，并在卡片上看到一致的备份状态与按钮可用性。

## 4. 信息架构

### 4.1 顶部导航

- 保留现有 `新建 / 保存 / 加载项目` 按钮
- 三个项目按钮整体左移，为右侧头像预留空间
- 新增圆形头像按钮：
  - 未登录：显示默认头像 / 图标
  - 已登录：显示昵称或邮箱首字母
  - 点击行为：
    - 未登录：打开登录 / 注册模态框
    - 已登录：切换到「个人中心」

### 4.2 个人中心一级标签

- 一级标签名称：`个人中心`
- 风格与现有一级标签一致
- 切换后进入二级标签区，整体风格复用元件管理页的二级标签样式

### 4.3 个人中心二级标签

#### A. 账号设置

- 未登录时优先显示类似移动端 App 的登录页，而不是配置卡片
- 登录页包含：
  - 登录 / 注册模式切换
  - 邮箱密码输入
  - OAuth 方式占位按钮（首版仅展示，不实现实际登录）
  - 用户协议 / 隐私政策勾选
- 已登录后切换为账号状态与设置视图
- 展示当前账号头像、昵称、邮箱、角色、登录方式
- 提供退出登录与切换账号操作
- 明确提示：若项目启用邮箱确认，则需要前往邮箱激活

#### B. 我的项目

- 读取系统设置中的 `项目文件夹路径`
- 将该目录下的项目文件夹以卡片形式展示
- 卡片字段（当前）：
  - 项目名、简介、元件数量、连线数量、最后修改时间
  - **备份状态**（无备份 / 已备份及时间）；已登录时与 Supabase **`project_backups`** + Storage 对齐
  - 操作：**上传备份** / **更新备份**、**撤销备份**、**打开项目**；若仅云端存在同名备份则展示「仅云端」卡片与 **下载备份**
- **云端备份规则（当前实现）**：
  - 单方案备份总大小上限 **5MB**；每用户最多 **10** 个不同 `project_key` 的备份方案（**更新已有方案**不计入新增计数）
  - 对象存储使用安全文件名 + **`__manifest__.json`** 记录原始相对路径，下载时按清单还原
  - 客户端对 Windows 路径做 **`/` 归一化**，与列表、删除、展示键一致
- **撤销备份**：需数据库 **`DELETE`** 策略（如 **`project_backups_delete_own`**：`authenticated` 且 `user_id = auth.uid()`）；主进程删除表记录成功后清理 Storage，并对删除行数做校验，避免「无权限仍提示成功」
- **交互**：上传与撤销在执行中展示与上传按钮一致的 **加载态**（转圈 + 背景条）；列表刷新后撤销按钮在无备份时应为 **禁用**
- 后续可扩展：缩略图封面、一键上传到创客集市、与分享长图导出联动

#### C. 创作中心

- 展示用户未来在创客集市中的数据入口
- 首版只做只读骨架：
  - 已上传作品数
  - 获赞数
  - 收藏数
  - 复刻数
- 预留后续列表区：
  - 我上传的作品
  - 待审核作品
  - 被驳回作品

#### D. 社区管理（仅管理员）

- 仅在 `profiles.role = 'admin'` 时显示
- 入口位置：`个人中心 -> 社区管理`
- 首版先保留 UI 骨架，后续承接审核列表、通过 / 驳回、审核记录等能力

## 5. 鉴权与技术路线

### 5.1 首版方案

- 账号系统基于 **Supabase Auth**
- 本轮优先使用 **邮箱 + 密码**
- 渲染层不直接持有复杂鉴权逻辑，统一通过 `preload -> ipc -> main process` 调用
- Supabase 会话在主进程持久化到本地文件，保证重启后可恢复

### 5.2 后续扩展

- 第二阶段接入 GitHub OAuth
- 第三阶段补充 Gitee 或其他国内常用 OAuth
- 如官方 Provider 不满足需求，再评估 `simple-oauth2`

### 5.3 安全要求

- 仅使用 `publishable key`，不在客户端保存 service role
- 不在普通日志中打印 access token / refresh token
- 主进程统一处理登录与登出，渲染层只拿最小状态快照
- 会话失效时 UI 自动回退匿名态

## 6. 数据模型

### 6.1 `profiles`

- `id`：对齐 `auth.users.id`
- `display_name`
- `avatar_url`
- `role`：`user` / `admin`
- `provider`：首版为 `email`，后续可扩展 `github` / `gitee`
- `created_at`
- `updated_at`

### 6.2 客户端状态

- `authStatus`：`anonymous` | `authenticated` | `expired`
- `currentUser`：最小用户快照
- `sessionPersistence`：本地恢复成功 / 失败

### 6.3 与社区联动的数据

- `posts`
- `post_likes`
- `post_favorites`
- `daily_uploads`

以上结构与 `9-maker-marketplace_prd.md` 保持一致。

### 6.4 项目云端备份（`project_backups` + Storage）

- **表 `public.project_backups`（示例字段）**：`user_id`、`project_key`（由规范化项目路径派生的稳定键）、`project_name`、`project_path`（规范化路径）、`backup_at`、`file_count`、`last_modified` 等
- **存储桶**：与主进程常量一致（如 `project-backups`），对象路径包含 `user_id` / `project_key` / 安全文件名及 **`__manifest__.json`**
- **RLS**：除读写自身行外，**必须**为已登录用户提供 **`DELETE`**（例如策略名 **`project_backups_delete_own`**），否则撤销备份在客户端会表现为无删除权限或状态不刷新

## 8. 交互细节

### 8.1 登录页

- 登录页字段：
  - 昵称（注册时可选）
  - 邮箱
  - 密码
  - 协议勾选
- 登录页操作：
  - 登录 / 注册模式切换
  - 邮箱密码提交
  - OAuth 方式占位展示
- 错误提示直接展示在页面内，并同时用应用通知提示

### 8.2 账号设置页

- 页面内保留一套同样的注册 / 登录表单
- 目的：
  - 用户进入个人中心后可继续完成注册
  - 避免只能依赖顶部弹窗
- 若已登录，突出显示账号摘要与退出按钮

### 8.3 我的项目卡片

- **打开项目**：从卡片跳转电路搭建并加载对应目录（已实现）
- 路径读取稳定、信息摘要清晰；备份相关操作经 **`preload` → IPC → `supabase/auth-service`**
- 分享入口可为占位提示；云端备份状态与按钮 **`disabled`** 与列表数据一致，避免撤销成功后仍显示可点撤销

---

## 9. 风控与限制

1. 邮箱密码注册失败时展示明确错误，避免只显示通用失败。
2. 邮箱确认未完成时，需要清晰提示用户去邮箱完成激活。
3. 会话过期时自动回退匿名态，避免界面显示已登录但请求实际失败。
4. 每日上传限额逻辑由社区侧定义，账号侧只负责展示与登录态约束。

---

## 10. 非功能需求

- **可靠性**：`preload` 异常不应拖垮整页基础功能。
- **安全性**：鉴权能力通过主进程桥接，敏感信息不直接暴露到全局对象。
- **可维护性**：邮箱密码与 OAuth 共用同一套 `profiles` / session / UI 状态结构。
- **可扩展性**：后续接入创客集市时不需要重做个人中心信息架构。

## 12. 验收标准

1. `preload` 正常加载后，渲染层可稳定访问 `window.electronAPI`。
2. 未登录时，点击右上角头像能打开注册 / 登录窗口。
3. 邮箱密码注册、登录、登出链路可通过主进程 IPC 调用 Supabase。
4. 个人中心可在未登录时显示完整登录页，登录后显示当前账号信息与角色信息。
5. **我的项目**页能读取系统设置中的项目目录并展示本地项目卡片。
6. 创作中心页已具备后续社区数据接入所需的展示骨架。
7. 管理员登录后，个人中心下可额外显示 `社区管理` 二级标签页。
8. 已登录用户可在 **我的项目** 中完成备份上传/更新、撤销（含 DB 删除校验与 Storage 清理）与下载恢复；无备份时撤销按钮为禁用；备份状态与列表刷新一致。
